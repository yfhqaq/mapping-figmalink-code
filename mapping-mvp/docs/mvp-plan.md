# MVP 方案：Figma 组件指纹映射到前端组件

## 背景
Figma 的业务页面基于 UX 组件库搭建，代码侧使用前端组件库（React + Ant Design）。
当前 AI 生成代码在组件选择上不够准确，尤其在不同业务文案和字段变化时容易选错组件。

## 目标（MVP）
建立一个最小可用且稳定的映射机制，让 LLM 能基于 Figma 组件“指纹”准确选取
前端组件并生成正确的 props。

> 指纹是什么：Figma 的 `INSTANCE` 节点会携带其来源组件的 `componentId`（或 `componentSetId`）。
> 这是组件库实例的“来源标识”，不受文案和业务字段变化影响。
>

### 范围内
- 手工维护一小部分核心组件的映射（Table / Upload / Tabs / SidebarNav / Button）。
- 基于 `componentId` / `componentSetId` 的指纹匹配。
- 只产出本地 LLM 上下文文件（无需发布到 Figma Dev Mode）。

### 范围外
- 映射自动化全覆盖。
- 纯视觉相似度的匹配（不依赖组件指纹）。

> 前期技术替代方案（在 UX 尚未完善组件库时）：
> 1) 结构回退匹配：当缺失指纹时，根据“结构块”回退识别（如 Head/Content/Footer -> Modal）。
> 2) 关键词补偿：用节点命名规则（如“筛选/查询/操作区”）触发复合规则。
> 3) 组件弱匹配：根据布局特征/子节点组合估计候选组件，输出低置信度结果。
> 这些方法只用于过渡期；长期仍以 UX 组件库指纹为准。

## 为什么可行
业务页面通过 Figma 组件实例复用 UX 组件库。即使文案/字段/数据不同，
实例节点仍保留来源组件指纹（`componentId` / `componentSetId`），因此：
1) **稳定性强**：文案改动不影响指纹；
2) **一致性高**：同一 UX 组件在不同页面具备相同指纹；
3) **可扩展**：新增组件只需补映射规则，不影响已存在结果。

## 技术选型
- Figma MCP：拉取页面节点与组件指纹。
- Code Connect 风格映射文件（`*.figma.tsx`）：作为映射源数据。
- LLM 上下文汇总文件（`llm-context.md`）：给 LLM 消费。
- 本地指纹缓存（`cache/component-index.json`）：避免重复拉取。

## 价值与收益
- 准确率提升：组件选择基于指纹而非文本。
- 成本更低：MVP 以手工映射为主，避免过早自动化投入。
- 可扩展：覆盖范围可随组件库增长逐步扩大。
- 可复用：同一映射可供多个 LLM 任务使用。

## 投入评估
- 初期：1-2 名工程师，2-3 天完成核心组件映射与上下文生成。
- 持续维护：UX 组件变更时更新映射，按需扩展组件。
- 无需新增基础设施。

## 实施路径
### 阶段 1：MVP（手工）
1. 选取 5-10 个核心 UX 组件。
2. 编写映射文件（`src/figma/*.figma.tsx`）。
3. 在 `figma.config.json` 填入节点链接。
4. 生成 LLM 上下文文件（`llm-context.md`）。
5. 用真实业务页面验证并输出匹配报告。

### 阶段 2：半自动化
1. 构建组件指纹缓存。
2. 结合 MCP + 缓存自动生成匹配报告。
3. 逐步加入“复合规则”（多组件 → 单组件 props/slots）。

### 阶段 3：规模化
1. 扩展到全部 UX 组件。
2. 建立映射版本治理机制。
3. 接入 CI 检测映射漂移。

## MVP 处理流程（面向非技术同学）
1. **拿到业务页面的 Figma 链接**  
   只需要页面节点链接，不需要整份设计文件。
2. **用 MCP 把页面结构“读出来”**  
   MCP 会返回页面中每个节点的类型、名称、是否为组件实例，以及它引用的 `componentId`。
3. **识别“组件指纹”**  
   只看 `INSTANCE` 节点。每个实例都会带上它来自哪个组件库组件的 `componentId` / `componentSetId`，
   这就是我们说的“指纹”。
4. **指纹 -> UX 组件**  
   用本地缓存或映射表把 `componentId` 翻译成 UX 组件名称（例如 `.Tab元素`、`.单元格-表体`）。
5. **UX 组件 -> 前端组件**  
   查 `*.figma.tsx` 映射规则，得到前端组件名与 props 的对应关系。
6. **处理“复合结构”**  
   如果页面里出现一组组件组合（如筛选器 + 表头 + 表体 + 按钮），
   触发复合规则，输出一个 Table 组件，并把这些区块映射到 `search / columns / toolBarRender`。
7. **生成结果与报告**  
   输出匹配结果（包含指纹来源、映射规则、置信度），并与“未使用映射”的输出做对比，
   让你能直接看到效果差异。

可优化点（现有方案基础上的增强）：
- 增加本地指纹缓存：减少重复拉取。
- 引入“组件缺口清单”：定位 UX 库缺失的高优先级组件。
- 提供“复合规则模板库”：减少每个新场景的配置成本。

## 关键难点与对策
1. **UX 组件库不完整**
   - 对策：补齐组件或引入复合规则。
2. **业务页面未使用组件实例**
   - 对策：推动 UX 侧使用组件库实例。
3. **变体/属性不对齐**
   - 对策：约定命名规范，定期同步映射。
4. **版本漂移**
   - 对策：缓存 + 定期更新复查。

## 成功标准
- 试点页面组件识别准确率 ≥ 80%。
- 新增组件的映射维护时间 < 1 小时。
- LLM 输出与实际使用组件一致，人工修正最小化。

## 下一步
- 确认试点组件清单。
- 提供官方 UX 组件链接。
- 用真实页面进行试点验证并评审报告。
- 建设“对比报告”机制（必须项）：
  - 统一报告格式：包含指纹来源、匹配结果、置信度、差异对比。
  - 同页面双输出：启用映射 vs 未启用映射。
  - 可视化对比：让非技术同事也能直观看差异。
- 最小侵入接入策略：
  - 不改现有代码生成流程，只增加“映射上下文”输入。
  - 把映射文件作为独立仓库或独立目录维护，便于团队共享。
  - 提供一键生成报告脚本，降低同事使用成本。
- 上下文裁剪计划（后续阶段）：
  - 默认只提供精简版映射（如 `mapping-llm.json`），只保留组件名、props 映射、业务 snippet、scope。
  - 按需补充字典（`mapping-llm.dictionary.json`），仅在模型出现“字段语义不明”时追加。
  - 定向裁剪：根据业务页面节点，仅抽取“命中的组件 + 相关业务映射片段”，不全量投喂。
  - 分层输入：先给精简版 → 需要时给字典 → 仍不足再给局部 `code-connect.json` 片段。
  - 控制窗口：同一请求中上下文不超过 N 条组件映射（避免注意力被稀释）。
  - 记录增量：仅追加“本次页面新增/变更的映射”，减少重复上下文。
  - 裁剪只作用于“上下文展示”：内部全量匹配，外部只给裁剪后的上下文，确保匹配正确性。
  - 设“最小保留集合”：命中组件、相邻复合规则、业务映射片段必须保留。
  - 兜底策略：若 LLM 找不到组件，自动补发更大窗口或局部原始 JSON。
- 建设“对比报告”机制（必须项）：
  - 统一报告格式：包含指纹来源、匹配结果、置信度、差异对比。
  - 同页面双输出：启用映射 vs 未启用映射。
  - 可视化对比：让非技术同事也能直观看差异。
- 最小侵入接入策略：
  - 不改现有代码生成流程，只增加“映射上下文”输入。
  - 把映射文件作为独立仓库或独立目录维护，便于团队共享。
  - 提供一键生成报告脚本，降低同事使用成本。
- 后续优化（可选）：
  - 索引摘要 + 向量检索：
    - 目标：在不牺牲确定性匹配的前提下，用向量召回补齐上下文。
    - 索引内容：组件摘要、使用场景、典型 props/slots、业务映射片段、scope；指纹与组件 ID 作为结构化字段做精确过滤。
    - 检索策略：先规则通道命中 → 再向量 TopK 补充；向量只做“补充召回”，不替代规则通道。
    - 粒度分层：组件级 / 复合规则级 / 业务片段级，避免大块文本稀释语义。
    - 组装流程：过滤候选 → 向量召回 → rerank → 组装“最小保留集合 + TopK 补充”上下文。
    - 评估与可观测：记录召回来源（规则/向量）、TopK 列表、命中原因，便于对比报告验证。
