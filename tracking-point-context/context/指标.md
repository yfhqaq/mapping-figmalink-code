指标
统计方式
释义

活跃用户数
distinct(user_id)
系统整体的使用情况，反映有多少用户访问或使用了系统，不区分具体功能
1. 低活跃用户数 → 系统或功能可能没人用
2. 判断整体系统是否被用起来
覆盖率
（目标用户覆盖率）

对目标用户覆盖了多少
1. 功能是否被感知
功能使用用户
（使用活跃用户数）
distinct(user_id where event)
使用某功能的独立用户数
1. 功能使用人数很少 → 可能是伪需求或功能不清楚
2. 判断功能是否被真正使用
功能覆盖率
（功能使用率）
功能 UV / 总 UV
功能使用人数占活跃用户总数比例

1. 覆盖率低 → 功能可能不吸引人，或入口不明显
2. 判断功能在系统中是否发挥价值
使用频次
（人均使用次数）
count(event) / user
单用户在统计周期内使用该功能的平均次数

1. 使用频次低 → 功能可能不重要，或流程设计不合理
2. 判断功能粘性及价值

路径发起量

distinct(trace_id)
用户在统计周期内发起某条业务路径（操作流程/事件链）的次数，代表用户尝试进入该流程的数量
1. 发起量低 → 用户没有尝试该流程，可能路径入口不明显或兴趣不高
2. 判断功能或流程是否被用户触达
路径完成率
完成路径 / 发起路径
用户成功走完整条路径的比例，衡量路径的可达性和用户执行成功率
1. 完成率低 → 路径设计可能复杂、阻碍多
2. 判断用户在流程中是否能顺利完成目标

step_n / step_(n-1)

1. 转化率低 → 某步骤可能存在体验或功能问题
2. 精细定位路径中的瓶颈或流失点

页面停留时间
page_leave - page_view
除以访问量 = 平均页面停留时间
用户从进入页面到离开页面所消耗的时间，包含页面加载、信息浏览及操作时间。
停留时间过短：可能页面无价值或误入；
过长：需结合性能指标判断是系统慢，还是信息复杂、操作成本高。
操作间隔

click_n - click_(n-1)
除以点击量=平均操作事件间隔
用户在同一页面或同一流程中，两次连续操作之间的时间间隔。
间隔过长：用户在思考、找不到入口或页面信息理解成本高；
间隔过短且频繁：可能是误触或重复点击。
重复操作率
重复 click / session
从路径的折线图里能看出来
在一次会话中，用户对同一元素或同一操作的重复点击比例。
偏高：强烈信号，通常意味着接口慢、反馈不明确或操作未生效；
是判断“系统慢 vs 体验差”的关键指标。
回退率

上一步点击率

用户进入页面后，上一步点击率

回退率高：页面不符合预期、路径设计不合理或信息层级混乱；
常用于发现路径断点和错误入口。
曝光点击率

click / expose

页面或模块中，被曝光的元素实际被点击的比例。
低 CTR：曝光有效但不吸引，可能文案、位置或视觉权重不足；
用于评估模块和功能的真实价值。
性能与稳定
慢是不是系统问题？
指标
统计方式
释义
判断意义
页面加载耗时
【先用arms】
page_perf.duration
从用户发起页面访问到页面完成渲染（可交互）的总耗时，包含网络、接口请求、资源加载和前端渲染时间。
衡量用户直观体验的第一指标；
整体偏高说明首屏慢、页面重；
需结合 P95 判断是否“少数人特别慢”。
接口耗时
【先用arms】
api_perf.duration
接口从请求发起到返回响应的耗时，包含网络传输和后端处理时间。
判断系统处理能力和稳定性；
接口慢往往直接导致页面慢和重复操作；
是性能优化最直接的抓手。
P95 / P99
【先用arms】
percentile
在所有请求中，95%（或 99%）请求的耗时不超过该值，用于衡量长尾性能。
比平均值更能反映真实体验；
P95 高：大量用户体验差；
P99 高：极端慢请求，常由异常或资源争用导致。
慢请求占比
【先用arms】
duration > threshold
请求耗时超过预设阈值的比例，用于衡量性能是否达标。
用于日常监控和告警；
占比升高通常是系统负载、外部依赖或代码回退问题；
比“平均耗时”更稳定。
错误率
【先用arms】
error / total
请求返回错误（接口异常、超时、前端异常等）的比例。
直接影响业务成功率；
错误率升高往往比慢更致命；
是性能与稳定性之间的“红线指标”。

平均完成耗时

path_end - path_start

用户从路径起点到完成终点所经历的平均总耗时，包含系统处理时间与用户操作时间。
衡量流程整体效率的基础指标；
整体偏高说明流程复杂或存在系统瓶颈；
需结合步骤耗时拆解判断“慢在系统还是慢在人”。
P95 完成耗时
【先不做】
路径 P95

95% 的路径在该耗时内完成，用于衡量路径完成效率的稳定性和长尾体验。
判断是否存在大量体验差用户；
P95 偏高往往意味着关键步骤存在性能抖动或偶发异常；
比平均值更适合做优化目标。
人效
提效类需求【会话事件提升了多少】从0-1类需求【总的完成单量 * 平均会话时间】对比用户线下操作一单完成的时间
完成量 / 活跃用户
在一定周期内，平均每个活跃用户完成的业务量，反映系统对人的效率支撑能力。
连接系统效率与人力成本的核心指标；
人效下降通常与路径变慢、流程变复杂或系统不稳定相关；
是判断系统是否“拖慢人”的直接证据。
慢路径完成率
【先不做】
slow vs normal
在慢路径（耗时超过阈值）与正常路径中，成功完成业务的比例对比。
衡量性能问题是否真正影响业务结果；
若慢路径完成率显著低于正常路径，说明性能已造成用户放弃或失败。
性能损失业务量
【先不做】
slow/error × biz_id
因性能问题（慢或错误）导致未能顺利完成的业务对象数量。
将性能问题量化为“业务损失规模”；
用于识别哪些业务因性能问题被直接影响，是排优先级的重要依据。
性能影响金额
【先不做】
失败 × biz_amount
因性能异常或失败导致的业务金额损失总和。
性能问题的最高级指标；
可直接用于评估优化 ROI 和向管理层说明“为什么要优先做性能”。
