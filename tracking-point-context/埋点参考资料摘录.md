## 1. 字节跳动：企业级埋点采集方案（规划 + 设计 + 推广）

来源：
- [如何设计企业级数据埋点采集方案？](https://www.cnblogs.com/bytedata/p/16695824.html)

### 关键可落地点
- **先定义、后采集**：埋点设计文档要明确“触发条件、事件名、属性类型与取值示例”，保证口径统一、可追溯。
- **角色责任**：需求方、设计方、研发、测试、验收必须明确，避免“谁都能改、没人负责”。
- **事件属性 vs 用户属性**：
  - 用户属性是“最新状态”（如 VIP 等级），适合服务端维护；
  - 事件属性记录“当次发生时状态”（如支付金额、品类），用于精细拆分。
- **不要为了埋点而埋点**：重要性/成本评估后做取舍，MVP 重点抓关键路径和关键价值事件。

### 映射到我们的 MVP
- 事件字典里必须有：`owner / status / version / required_props / 口径说明`。
- Schema 分层：`user（画像快照）` 与 `event.props（当次属性）` 必须区分。

## 2. 人人都是产品经理：埋点管理系统（元信息 + 验收 + 追溯）

来源：
- [如何设计出一个实用高效的埋点管理系统？](https://www.woshipm.com/data-analysis/5107308.html)

### 关键可落地点
- **元信息集中**：事件/属性散落在不同人手里会导致沟通成本极高。
- **验收难**：非可视化验收成本高，需要“实时回显/抓包工具”。
- **版本管理**：埋点异常要能追溯到“哪个版本/哪次需求/谁改的”。
- **数据校验**：字段错、事件名错要能被系统发现，而不是事后清洗。

### 映射到我们的 MVP
- 必做：事件字典 CRUD + 发布版本 + 校验开关（strict/lenient）+ 实时事件流（验收）。

## 3. 阿里云 ARMS RUM：会话/性能/自定义事件的工业做法（参考）

来源（你提供的文档链接）：
- `https://help.aliyun.com/document_detail/252716.html`
- `https://help.aliyun.com/document_detail/252718.html`
- `https://help.aliyun.com/document_detail/327153.html`
- `https://help.aliyun.com/document_detail/399751.html`
- `https://help.aliyun.com/document_detail/2840363.html`

### 我们借鉴的关键点
- **会话（Session Trace）**：用 `session_id` 把一段时间内的行为串起来；
- **性能事件是第一类公民**：页面性能、接口耗时、错误率与业务事件同一套链路 ID 关联；
- **自定义事件标准化**：为业务事件提供统一 API（init/config/custom 等概念）。

### 映射到我们的 MVP
- Schema 必须同时容纳：`event（业务/交互）` + `perf（性能/错误）` + `session/trace（串联）`。
- L5 价值计算必须依赖：`biz_id/biz.amount` 与 `perf.api.is_slow / perf.api.status_code / perf.error` 的关联。

## 4. 你们的《指标.md》：最终验收口径

来源：`context/指标.md`

### MVP 的验收优先级建议（只做最关键）
- **L2 路径（能不能走完）**：path_start/path_step/path_end（或 success 事件）
- **L4 性能（系统拖不拖后腿）**：api_perf + page_perf
- **L5 价值（丢了多少钱）**：slow/error × biz_id / 失败 × biz_amount
